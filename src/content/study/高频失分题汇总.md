## 🔴 第一梯队：多场翻车、一票否决级

### Q1：Handler 消息机制 —— Looper.loop() 为什么不卡死主线程？底层机制是什么？
- **被问次数**：懂车帝一面、TikTok、今日头条
- **你当时的回答**：「如果没有消息它就会触发休眠」—— 停在了表面描述，**epoll / nativePollOnce 一个词没提**
- **面试官信号**：懂车帝一面面试官连续追问 postDelayed、ThreadLocal、同步屏障、IdleHandler，试探你的深度上限，全部命中盲区
- [ ] **需要掌握的答题链**：消息队列为空 → `nativePollOnce()` → Linux `epoll_wait` 阻塞（释放 CPU）→ 新消息 → `nativeWake()` → `eventfd` 写入唤醒 → Looper 继续循环
- **学习文件**：[01-Handler-View绘制-事件分发.md](/study/01-Handler-View%E7%BB%98%E5%88%B6-%E4%BA%8B%E4%BB%B6%E5%88%86%E5%8F%91/)

### Q2：LiveData setValue vs postValue 区别？LiveData 的粘性事件问题？
- **被问次数**：懂车帝一面、TikTok、小红书、今日头条
- **你当时的回答**：懂车帝一面直接说「**没了解过**」
- **面试官信号**：这是 Android 开发的送分题，直接暴露 Java 基础薄弱
- [ ] **需要掌握的要点**：
  - `setValue`：必须主线程调用，直接更新 value 并同步通知 Observer
  - `postValue`：可在子线程调用，内部通过 Handler 切到主线程再调 setValue；连续多次 postValue 只保留最后一次
  - 粘性事件：新注册的 Observer 会立刻收到最后一次 setValue 的值。解法：用 `SharedFlow(replay=0)` 替代，或者 SingleLiveEvent 模式

---

### Q3：StateFlow vs SharedFlow vs LiveData 完整对比
- **被问次数**：懂车帝、小红书、TikTok、今日头条（**4 场考、4 场弱**——你的最顽固弱点）
- **你当时的回答**：准备版能讲清楚，但面试现场表达不流利
- [ ] **30 秒标准答案**：
  - StateFlow：热流、必须有初始值、内置 `distinctUntilChanged` 去重、新订阅者立刻收到最新值 → **适合 UI State**
  - SharedFlow：无初始值、可设 replay/背压策略 → **适合一次性事件**（Toast / 导航）
  - LiveData：自动生命周期感知、setValue 需主线程 → **纯 Java 项目或简单场景**
  - 选择策略：UI 状态 → StateFlow；一次性事件 → SharedFlow(replay=0)；纯 Java → LiveData

---

### Q4：RecyclerView 四级缓存说不全
- **被问次数**：懂车帝一面、TikTok、字节
- **你当时的回答**：只说出 2 层（屏幕内 + 屏幕外），**ViewCacheExtension 漏掉**，ViewType 承认没设过
- [ ] **需要掌握的完整链**：
  1. `mAttachedScrap` — 屏幕内临时缓存，layout 过程复用
  2. `mCachedViews` — 刚滑出屏幕（默认 2 个），数据没清空，零开销复用
  3. `mViewCacheExtension` — 自定义缓存（极少用）
  4. `RecycledViewPool` — 按 viewType 共享，数据清空需 rebind
- **学习文件**：[03-RecyclerView.md](/study/03-RecyclerView/)

## 🟡 第二梯队：高频被问、回答不完整

### Q6：MVP 架构 —— 直接说「没用过」
- **被问次数**：今日头条、TikTok
- **你当时的回答**：今日头条直接说「MVP 这块我还没用过」
- **面试官反馈原话**：「可能得稍微加强一下那个技术视野。不能说只聚焦于你做的东西。」
- [ ] **30 秒标准答案**：MVP = View-Presenter-Model。Presenter 持有 View 引用，接口契约多、耦合重，容易内存泄漏。相比之下 MVVM 的 ViewModel 不持有 View 引用更安全，MVI 进一步用单向数据流解决了状态分散问题。
- **学习文件**：[05-MVVM-MVP-MVC对比.md](/study/05-MVVM-MVP-MVC%E5%AF%B9%E6%AF%94/)

---

### Q7：协程底层原理 —— 没提 Continuation / CPS 变换
- **被问次数**：懂车帝二面、TikTok、字节
- **你当时的回答**：讲了挂起/恢复/线程调度，但**没提 Continuation 和状态机**
- [ ] **需要掌握的要点**：
  - `suspend` 函数编译后多一个 `Continuation` 参数（CPS 变换）
  - Continuation 本质是一个状态机，每个挂起点对应一个 `label` 分支
  - 挂起时：保存当前状态到 Continuation，返回 `COROUTINE_SUSPENDED` 标记，释放线程
  - 恢复时：Dispatcher 调度到目标线程，调用 `Continuation.resumeWith()` 从上次的 label 继续执行

---

### Q8：Fragment 生命周期 + viewLifecycleOwner
- **被问次数**：今日头条、小红书（**2 场被面试官直接点名「最基础的东西」**）
- [ ] **核心要点**：
  - `onDestroyView ≠ Fragment 销毁`。View 销毁了但 Fragment 实例还活着
  - LiveData/Flow 订阅必须用 `viewLifecycleOwner` 而不是 `this`，否则 View 已销毁但 Observer 还在收数据 → 崩溃
  - ViewPager2 + Fragment：切换 Tab 时当前 Fragment 走 onPause → onStop，不走 onDestroyView

---

### Q9：Java 四种引用类型 + 单例模式手写
- **被问次数**：影石、小红书、今日头条（**三连暴露，送分题级别**）
- **你当时的回答**：影石面试官反馈「Java 基础偏薄」
- [ ] **四种引用**：
  - 强引用：永不回收（OOM 也不收）
  - 软引用 SoftReference：内存不足时回收 → 图片缓存
  - 弱引用 WeakReference：下一次 GC 就回收 → LeakCanary 检测泄漏
  - 虚引用 PhantomReference：get() 永远 null → 管理堆外内存
- [ ] **单例手写**：
  - Java：双重检查锁 + volatile（防指令重排）
  - Kotlin：直接用 `object` 声明

### Q10：图片/视频编解码渲染管线 —— 方向答偏
- **被问次数**：懂车帝二面
- **你当时的回答**：把编解码管线答成了 View 的 measure/layout/draw —— **面试官直接说「底层编解码不是特别清楚是吧？」**
- [ ] **正确的链路**：
  - 图片：压缩格式（JPEG/WebP）→ 解码器（BitmapFactory / Skia）→ Bitmap 内存分配 → Canvas 绘制 → GPU 渲染到 Surface → SurfaceFlinger 合成显示
  - 视频：H.264 容器 → MediaCodec 硬解 / FFmpeg 软解 → Surface 渲染 → 音画同步（PTS）

## 🟢 第三梯队：偶发被问、有改善空间

### Q12：Gradle 多仓库 AAR/源码切换原理
- **被问次数**：今日头条
- **面试官反馈原话**：「你其实并不太清楚它其中的原理」
- [ ] **核心原理**：Gradle 的 `dependencySubstitution` 机制。在 configuration 阶段，根据配置用 `substitute(module(...)).using(project(...))` 把远程 AAR 依赖替换为本地 project 依赖

---

### Q13：IdleHandler / 同步屏障 / ThreadLocal
- **被问次数**：懂车帝一面（连续 3 个「没听过」）
- [ ] **IdleHandler**：MessageQueue 空闲时触发的回调，常用于延迟初始化
- [ ] **同步屏障**：target 为 null 的 Message，让 Looper 跳过同步消息只处理异步消息，View 绘制的 `scheduleTraversals` 用它保证绘制优先级
- [ ] **ThreadLocal**：每个线程持有独立副本。Looper.myLooper() 就是从 ThreadLocal 取的

---

### Q14：Compose 基础 —— 「了解」怎么说
- **被问次数**：懂车帝、影石
- [ ] **标准话术**：「了解基本概念但没有在商业项目中用过。核心区别是声明式 vs 命令式——View 体系是告诉系统"怎么做"，Compose 是告诉系统"UI 应该长什么样"，状态变了 Composable 函数重新执行，框架自己算 diff 更新。另外 Compose 跳过 View 体系直接操作 Canvas，不存在嵌套层级问题。」

---

### Q15：简历条目缺少「原理层」—— 只会说「怎么用」不会说「为什么」
- **被问次数**：今日头条（被现场抓包）
- **面试官追问模式**：「你说多仓库 AR/源码切换——它是怎么选择源码还是 AR 的？Gradle configure 阶段怎么确保该走哪条路径？」
- [ ] **自检方法**：简历上的每一条，必须能回答三层——「是什么」→「怎么做」→「底层原理是什么」
