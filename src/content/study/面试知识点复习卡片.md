## 知识点 1：Handler/Looper 消息机制 + View 绘制三大流程 + 事件分发

### 1. 核心原理讲解
*   **Handler 消息机制**：Android 的事件驱动核心。主要由 `Handler`、`Looper`、`MessageQueue` 和 `Message` 协作完成。`Looper.loop()` 内部是一个死循环，不断从 `MessageQueue` 中读取消息交给 Handler 执行。主线程没有消息时不会卡死，是因为底层利用了 Linux 的 `epoll` 机制。在无消息时，主线程会释放 CPU 资源并进入挂起状态（阻塞在 `epoll_wait`），当有新消息写入管道时会被重新唤醒，因此不占用 CPU 资源。
*   **View 绘制流程**：`measure`、`layout`、`draw` 三大步骤。`measure` 根据父 View 传入的 `MeasureSpec`（包含 `EXACTLY` 精确值、`AT_MOST` 最大值、`UNSPECIFIED` 未指定）和子 View 自身的 `LayoutParams` 确定测量宽高；`layout` 根据测量宽高确定 View 在父容器中的四个顶点位置（L, T, R, B）；`draw` 负责将 View 绘制到屏幕上，包含绘制背景、自身、子 View 以及装饰（如滚动条）。
*   **事件分发机制**：触摸事件按照 `Activity -> PhoneWindow -> DecorView -> ViewGroup -> View` 链条分发。核心方法为 `dispatchTouchEvent()`（负责分发）、`onInterceptTouchEvent()`（ViewGroup 独有，负责拦截）、`onTouchEvent()`（负责消费）。多层嵌套的滑动冲突，通常利用外部拦截法（在父容器的 `onInterceptTouchEvent` 进行判断拦截）或内部拦截法（子 View 通过 `requestDisallowInterceptTouchEvent` 阻止父容器拦截）进行解决。

### 2. 与候选人经历的桥接话术
> “在我独立负责 AI 图创 Feed 流搭建时，由于涉及到多层嵌套滚动容器（如双列瀑布流与页面外层滑动组件），我大量接触并优化了复杂的事件分发与嵌套滑动冲突；同时为了优化局部刷新性能，我利用 RecyclerView Payload 机制避免了 Item 重复执行完整的 measure 与 layout 过程，保障了滑动流畅度。”

### 3. 3-5 个高频追问 + 参考答案要点
*   **追问 1：主线程 Looper.loop() 死循环既然被挂起，那么应用又是如何响应用户点击事件的？**
    *   *答案要点*：用户点击屏幕产生输入事件（InputEvent），系统底层的 `InputDispatcher` 通过 Binder 向应用进程的 `InputChannel` 发送消息，这会往主线程 of `MessageQueue` 中写入数据，从而唤醒阻塞在 `epoll_wait` 的 Looper，进入下一次循环来分发并处理该点击事件。
*   **追问 2：自定义 View 时，如果在 onMeasure 中直接使用 super.onMeasure(widthMeasureSpec, heightMeasureSpec)，对于 wrap_content 会有什么问题？**
    *   *答案要点*：如果不重写处理 `AT_MOST`（即 `wrap_content`），自定义 View 的宽高会默认被设置为父容器剩余的最大空间（等同于 `match_parent`）。因此，必须在 `onMeasure` 中针对 `MeasureSpec.getMode(spec) == MeasureSpec.AT_MOST` 的情况，为 View 指定一个默认的兜底大小。
*   **追问 3：在事件分发中，如果子 View 的 onTouchEvent 返回了 false，后续的 ACTION_MOVE 和 ACTION_UP 还会传递给它吗？**
    *   *答案要点*：不会。如果子 View 在 `ACTION_DOWN` 时 `onTouchEvent` 返回了 `false`，说明它不消费该事件序列。系统会将该 View 从事件接收列表中移除，后续的 `ACTION_MOVE` 和 `ACTION_UP` 将直接交由其父容器的 `onTouchEvent` 处理，不再向下分发。

### 4. 当前知识缺口提示
*   > [!WARNING]
    > **细节防坑**：面试官可能会问你：“当子 View 已经调用过 `requestDisallowInterceptTouchEvent(true)` 阻止拦截后，父 View 的 `ACTION_DOWN` 能否被拦截？”。答案是**绝对不能**。因为在 `dispatchTouchEvent` 分发 `ACTION_DOWN` 时，ViewGroup 会重置所有的拦截标志位（包括清除 disallow 状态）。因此，父 View 的 `ACTION_DOWN` 总是会被执行拦截判断，必须保证 `ACTION_DOWN` 时父 View 不拦截，事件才能到达子 View。
*   > [!NOTE]
    > **Compose 映射**：由于你没有 Compose 实战经验，如果被问及 Compose 绘制与测量，只需点出其底层依然依托 Android 系统的 Canvas 进行绘制，但其测量模型从 MeasureSpec 变成了 `Constraints` 约束向下传递、`Layout` 尺寸向上返回，只进行一次测量（Single Pass Measurement），从而避免了传统 View 树多次 measure 带来的性能损耗。

## 知识点 2：性能优化与分析工具

### 1. 核心原理讲解
*   **Android Profiler**：包含 CPU、Memory、Network、Energy 分析面板。CPU Profiler 通过方法采样（Sampled）或全量检测（Instrumented）记录耗时；Memory Profiler 提供实时的内存抖动视图与 Heap Dump 堆转储分析，能直观看到对象强引用关系。
*   **Perfetto / Systrace**：系统级性能分析利器，基于 Linux `ftrace` 机制。通过在系统和 App 的关键节点插桩（如 `Trace.beginSection()`），以图形化时间线方式展示 CPU 调度、渲染线程（RenderThread）状态、Binder 通信耗时以及掉帧（Jank）情况。
*   **LeakCanary 原理**：利用 Java 弱引用（`WeakReference`）与引用队列（`ReferenceQueue`）。当 Activity 或 Fragment 被销毁时，将其包装为弱引用并关联引用队列。系统 GC 后，如果该对象没有进入 `ReferenceQueue`，说明它仍被外部强引用持有。系统会判定可能发生泄漏，并在延迟后触发 `Heap Dump`，通过分析 `.hprof` 堆快照来找出泄露对象到 GC Root 的最短引用链。
*   **协程内存泄漏常见模式**：协程虽然不是真正的线程，但其底层的挂起状态是通过 `Continuation`（状态机）在堆中保存的。如果协程的 `CoroutineScope` 生命期超出外部组件（例如使用 `GlobalScope` 或未在 ViewModel 销毁时主动 cancel 子协程 `Job`），那么协程体内持有的外部类变量（如 Activity、ViewModel 引用）以及回调都会无法被垃圾回收，直接引发内存泄漏。

### 2. 与候选人经历的桥接话术
> “在针对 AI 图创项目进行稳定性治理时，我曾用 LeakCanary 配合 Android Profiler，定位并修复了一处由于使用 `suspendCancellableCoroutine` 进行 OkHttp 协程桥接时，未正确取消网络请求导致的 ViewModel 内存泄漏，并总结了一套针对异步回调转协程的防御性编码规范。”

### 3. 3-5 个高频追问 + 参考答案要点
*   **追问 1：请讲讲你排查 OkHttp + 协程内存泄漏的具体引用链、根因以及最后的代码级解决方案？（重点展现）**
    *   *答案要点*：
        *   **泄露引用链**：`OkHttp Call/Callback -> continuation -> CoroutineScope -> ViewModel`。
        *   **根因**：使用 `suspendCancellableCoroutine` 将 OkHttp 异步回调转为挂起函数。当用户退出页面、ViewModel 销毁时，`viewModelScope` 会被取消。但此时底层的 OkHttp 请求仍在飞，网络线程持有的 Callback 依然引用着挂起点的 `continuation`，导致整条引用链无法被 GC 回收。
        *   **解决方案**：
            1. 注册 `continuation.invokeOnCancellation { call.cancel() }`，在协程被取消时立即强行终止 OkHttp 请求；
            2. 在 Callback 的 `onResponse` 和 `onFailure` 回调入口处，增加 `if (!continuation.isActive) return` 的 `isActive` 状态守卫判定，避免向已取消的挂起点继续 resume 数据。
*   **追问 2：Systrace / Perfetto 中最常关注哪些指标来排查滑动掉帧问题？**
    *   *答案要点*：关注 `Choreographer#doFrame` 的耗时（是否超出 16.6ms/11.1ms）、主线程（UI Thread）与渲染线程（RenderThread）的 `Running` 状态是否断开，以及是否存在由于锁竞争或 Binder 调用导致的 `Blocked` 状态。
*   **追问 3：在没有 LeakCanary 的生产环境，你如何实现核心组件的内存泄漏监控？**
    *   *答案要点*：可以通过在 Activity/Fragment 销毁时，将其放入弱引用列表中。在后台线程定期触发轻量级的可达性检测（如利用 `WeakReference` 配合主动触发 `Runtime.getRuntime().gc()` 后判断引用是否释放）。若发生泄露，记录关键类名和时间并上报埋点，或通过自研的 Dump 裁剪工具将堆快照脱敏后回传分析。

### 4. 当前知识缺口提示
*   > [!IMPORTANT]
    > **细节防坑**：面试官可能会深挖：“在 `suspendCancellableCoroutine` 的 `invokeOnCancellation` 中调用 `call.cancel()`，这个取消回调到底是在哪个线程执行的？”。答案是：**它执行在触发协程取消（cancel）的那个线程**（例如主线程销毁 ViewModel时），这意味着你在该回调里不能做任何耗时的同步阻塞操作，`call.cancel()` 内部是非阻塞的，所以是安全的。
*   > [!NOTE]
    > **TraceView 的开销**：如果被问及传统 TraceView，一定要答出：TraceView 会对所有运行的方法强行插桩，导致性能开销极高，测出来的耗时数据通常严重失真（放大真实耗时）。在实战中我们更倾向于使用 Systrace 或在 Profiler 中使用 `Sampled`（抽样）模式来降低运行时开销。

## 知识点 3：RecyclerView

### 1. 核心原理讲解
*   **四级缓存机制**：RecyclerView 极致流畅滑动的基石。
    1.  `mAttachedScrap`：屏幕内临时缓存。仅在局部刷新（如 `notifyItemChanged` 且有 Payload）或重新布局时起作用，ViewHolder 结构未变，直接复用无需重新 bind。
    2.  `mCachedViews`：屏幕外一级缓存（默认容量为 2）。滑出屏幕的 ViewHolder 带着数据完整存放在此，若马上滑回，可直接取用，零开销复用（无需 create 也无需 bind）。
    3.  `mViewCacheExtension`：自定义缓存。通常不使用，由开发者按需实现。
    4.  `RecycledViewPool`：ViewHolder 共享池。按 `itemViewType` 存放被彻底格式化的 ViewHolder，从池里取出来复用时，必须重新调用 `onBindViewHolder` 重新绑定数据。
*   **DiffUtil 与 Payload 局部刷新**：`DiffUtil` 利用 Myers 差分算法计算新旧数据集的最小差异并生成刷新指令。结合 `getChangePayload(oldItem, newItem)` 方法计算被修改的具体属性包（Payload），使得 `Adapter` 能够仅对 Item 内的特定 View（如点赞数 TextView）执行局部刷新，省去了整个 Item 重新测量、布局和重绘的性能消耗。
*   **瀑布流常见崩溃分析**：在瀑布流（如 `StaggeredGridLayoutManager`）中，常见崩溃为 `IndexOutOfBoundsException: Inconsistency detected`。这通常是因为 Adapter 的数据源在异步线程被并发修改，或者数据刷新指令与 RecyclerView 的滑动布局计算并发布局，导致 LayoutManager 保存的列状态（span）与 Adapter 中的真实数据位置不匹配。

### 2. 与候选人经历的桥接话术
> “在我开发的 AI 图创 Feed 流瀑布流场景中，我通过重写 `StaggeredGridLayoutManager` 自研了 `BiSerialFlowLayoutManager`，拦截了快速切 Tab 导致的 `onDetachedFromWindow` span 缓存清空行为，成功将瀑布流高频崩溃率降到了 0；同时利用 DiffUtil 和 Payload 局部刷新，大幅降低了点赞及评论状态改变时 ViewHolder 重新绑定的开销。”

### 3. 3-5 个高频追问 + 参考答案要点
*   **追问 1：既然有了 RecycledViewPool，那 mCachedViews 的意义是什么？它们有什么根本区别？**
    *   *答案要点*：`mCachedViews` 相当于“带状态复用”，它保留了原先绑定的所有数据和 Position 关系，复用时直接上屏，开销为零；`RecycledViewPool` 则是“无状态复用”，里面的 ViewHolder 数据已被擦除，复用时必须经过 `onBindViewHolder`。此外，`mCachedViews` 是 RecyclerView 私有的，而 `RecycledViewPool` 可以被多个 RecyclerView 实例所共享。
*   **追问 2：为什么你的 BiSerialFlowLayoutManager 要重写 onDetachedFromWindow 并实现为空？原生的 StaggeredGridLayoutManager 到底在这里做了什么导致了崩溃？**
    *   *答案要点*：原生的 `StaggeredGridLayoutManager` 在 `onDetachedFromWindow` 中会主动调用 `invalidateSpanAssignments()` 彻底清除所有 Item 的 span 索引缓存。当使用 ViewPager2 快速切 Tab 时，Fragment 会被 Detach。如果在 Detach 的同时 RecyclerView 仍在滑动或处于并发布局阶段，且锚点（Anchor）计算仍然指向原来的 item，由于 span 缓存被清空，就会因无法正确分配列索引而抛出数组越界或不一致崩溃。通过重写为空，保持了 span 缓存，从而规避了该崩溃。
*   **追问 3：DiffUtil 的比对是在主线程还是子线程？大列表会有什么性能隐患？怎么解决？**
    *   *答案要点*：`DiffUtil.calculateDiff` 的时间复杂度最坏为 $O(N + D^2)$，当数据集较大（如超过数百条数据）时，在主线程执行比对会造成明显卡顿。应该使用 `AsyncListDiffer` 或 `ListAdapter`，将比对算法放到后台线程的协程中执行，计算完毕后再将更新指令派发回主线程更新 UI。

### 4. 当前知识缺口提示
*   > [!WARNING]
    > **细节防坑**：面试官常考：“如何防止 RecyclerView 在快速滑动时出现图片错位？”。你要能够答出：图片加载框架（如 Glide）会为 ImageView 设置 Tag 绑定当前请求。在 ViewHolder 复用时，`onBindViewHolder` 会触发新的图片请求，Glide 会先取消 ImageView 上的旧请求并将其占位图设为空，从而防止滑动中旧图片异步加载完成后错位显示。
*   > [!NOTE]
    > **数据同步的坑**：关于 Adapter 数据源的并发修改，如果在多协程异步更新数据源，而在更新后没有立刻调用 `notify` 刷新，此时用户正好发生了滑动，就会触发不一致崩溃。解决方法是：所有对列表数据的增删改，必须在同一个主线程中顺序提交，或者通过 `AsyncListDiffer` 统一管理。

## 知识点 4：Jetpack Lifecycle + ViewModel

### 1. 核心原理讲解
*   **Lifecycle 核心实现**：采用观察者模式。通过 `LifecycleRegistry`（生命周期注册表）管理状态。支持的 LifecycleOwner（如 Activity / Fragment）在自身生命周期发生变化时，会向 `LifecycleRegistry` 发送 `Event`，触发其内部状态机 `State` 的转移，并遍历分发给所有的 `LifecycleObserver`，使普通组件能够无侵入地感知宿主生命周期。
*   **ViewModel 存活与保留机制**：当 Activity 发生屏幕旋转或系统语言变更等配置改变（Configuration Changes）时，Activity 会被销毁重建。但在销毁前，系统会调用 `retainNonConfigurationInstances()` 将 `ViewModelStore` 暂存。当新 Activity 重建并初始化时，通过 `ViewModelProvider` 能够重新获取并复用同一个 `ViewModelStore` 实例，从而保证了 ViewModel 中的数据在配置变更后依然保留。只有当 Activity 真正调用了 `finish()` 结束生命周期时，系统才会调用 ViewModel 的 `onCleared()` 以释放资源。
*   **StateFlow/MVI 的订阅防漏**：在 MVI 架构中，UI 订阅单一数据源 `StateFlow`。若直接使用 `lifecycleScope.launch` 并在其中 collect，当 App 被切到后台（Activity 进入 `STOPPED` 状态）时，协程的收集器并不会自动停止接收数据，依旧会导致非必要的性能消耗甚至空指针风险。最佳实践是利用 `repeatOnLifecycle(Lifecycle.State.STARTED)` 或 `flowWithLifecycle`。它们能在生命周期低于指定状态时，彻底**取消**订阅协程并回收资源；当页面重新回到前台时，自动**重新启动**协程进行订阅。

### 2. 与候选人经历的桥接话术
> “我们在 AI 产品中采用了强化后的 MVI 状态管理模式。通过使用 `viewModelScope` 配合 `repeatOnLifecycle(Lifecycle.State.STARTED)` 来订阅不可变的 `UiState`，我确保了在复杂的图创 Feed 流中，当页面不可见时彻底释放所有的 UI 订阅及数据处理资源，这也是我们能做到 14 个版本状态 Bug 为零的关键架构设计。”

### 3. 3-5 个高频追问 + 参考答案要点
*   **追问 1：ViewModel 既然能在屏幕旋转时存活，那如果应用在后台被系统彻底回收（Process Recreation）了，ViewModel 还能存活吗？如何应对？**
    *   *答案要点*：不能存活。当进程被系统杀掉重建时，ViewModel 同样会被彻底销毁。为了处理这类深度的“数据恢复”，应在 ViewModel 中注入 `SavedStateHandle`。它能够利用系统底层的 `onSaveInstanceState(Bundle)` 机制，在进程意外死亡重建时恢复关键的配置或状态参数。
*   **追问 2：为什么推荐使用 repeatOnLifecycle 接口，而不是已经弃用的 launchWhenStarted？两者底层有何区别？**
    *   *答案要点*：`launchWhenStarted` 在生命周期降到 `STARTED` 以下时，仅仅是将协程**挂起**（suspend），底层数据流的生产者（Producer）依然在向缓冲区发送数据；而 `repeatOnLifecycle` 在离开 `STARTED` 状态时会直接**取消**（cancel）底层协程，彻底断开数据链条，在返回前台时再从头 collect，性能表现更好，更符合资源利用最优化原则。
*   **追问 3：ViewModel 为什么不能持有 View、Activity 或者是 Fragment 的引用？**
    *   *答案要点*：ViewModel 的生命周期一般长于 Activity/Fragment（如屏幕旋转时，旧 Activity 销毁但 ViewModel 仍存活）。如果 ViewModel 持有 Activity 的强引用，会导致旧 Activity 销毁后无法被 GC 回收，从而造成严重的内存泄漏。如果确实需要 Context，必须使用继承自 `AndroidViewModel` 的全局 `Application`。

### 4. 当前知识缺口提示
*   > [!IMPORTANT]
    > **细节防坑**：面试官如果细抠：“在 Activity 重建获取旧 ViewModel 的过程中，如果我们在 `onCreate` 里重复向同一个 ViewModel 中通过 `observe()` 注册观察者，会造成重复订阅吗？”。答案是：**不会**。因为 LiveData 和 LifecycleObserver 内部会在注册时校验观察者是否重复，并且 LiveData 在注册时会绑定当前的 LifecycleOwner，一旦宿主销毁重建，旧的订阅关系会自动解除，非常安全。
*   > [!NOTE]
    > **MVI 数据流控制**：MVI 风格中，UI 状态是一次性全量分发的。如果页面包含非常多的 UI 组件，任何一个小细节状态变化（如只改了一个 checkbox）都会分发一个新的 `UiState`。此时你需要知道如何用 Flow 的 `distinctUntilChanged()` 或者是属性局部映射方法，防止不相关的子 View 触发无效刷新。

## 知识点 5：MVVM / MVP / MVC 对比

### 1. 核心原理讲解
*   **MVC 模式**：Model-View-Controller。Controller 负责业务逻辑（通常由 Activity 承担），View（XML/View）负责展示。缺点是 View 和 Model 之间存在双向关联，耦合严重。且由于 Android 中 Activity 既承担了视图布局又承担了事件处理，导致 Activity 异常臃肿，难以进行单元测试。
*   **MVP 模式**：Model-View-Presenter。Presenter 作为中介，完全隔离了 View 和 Model。View（Activity/Fragment）不直接跟 Model 交互，所有的交互都通过 Presenter 接口定义的契约进行。缺点是接口定义繁琐（契约类代码暴增），且 Presenter 常期持有 View 的强引用，极易发生内存泄漏。
*   **MVVM 模式**：Model-View-ViewModel。ViewModel 不再持有 View 的任何引用，而是通过 LiveData、Flow 等响应式数据流将自身数据暴露。View 主动订阅数据流以驱动 UI 自动更新。彻底消除了 Presenter 中的大接口，极大地解耦了视图层 and 逻辑层，且由于 ViewModel 中没有任何 UI 及 Context 依赖，可以非常方便地在 JVM 上运行单元测试。
*   **MVI 风格强化**：在传统 MVVM 中，UI 的各个状态（如 isLoading、dataList、errorMsg）通常分散在多个 LiveData 中，容易出现多线程数据时序问题导致状态冲突（如同时显示 Loading 与错误页面）。MVI 提倡**单向数据流**（Unidirectional Data Flow）与**唯一可信数据源**。它将界面状态聚合为一个不可变的单一数据类（如 `UiState`），UI 只能通过发送动作 `Intent`（如点击加载、刷新）来改变这个 `UiState` 并全量接收新状态，从根本上消除了状态不一致的情况。

### 2. 与候选人经历的桥接话术
> “我们在项目中采用了 MVVM 的升级版——强化 MVI 单向数据流的状态设计。通过将 UI 状态统一收拢在一个不可变的密封类（sealed class）`UiState` 中，并利用 `StateFlow` 进行单向流转，我独立搭建的 AI Feed 流在 14 个版本的密集迭代中没有发生过一起由状态不一致引起的 UI 异常 Bug。”

### 3. 3-5 个高频追问 + 参考答案要点
*   **追问 1：既然 MVI 能杜绝状态不一致，那为什么没有在所有的 Android 项目中彻底取代 MVVM？它有什么短板？**
    *   *答案要点*：MVI 存在以下不足：
        1.  **内存与 GC 开销**：因为每次哪怕只有极小的状态改变，也必须创建并拷贝（`copy`）一个新的 `UiState` 实体，高频更新时可能会给 JVM 带来频繁 GC 的压力。
        2.  **代码模板冗长**：需要额外定义大量的 Intent、Action 和 State 组合，对于简单业务显得有些过度设计。
        3.  **UI 刷新颗粒度过大**：若子 View 没有做好防重刷新机制，全量 State 更新可能会导致界面不必要的重绘。
*   **追问 2：MVI 中的 "Intent" 与 Android 系统的 "Intent" 有什么区别？**
    *   *答案要点*：两者完全没有任何关系。Android 系统的 `Intent` 是组件间通讯的信使（启动 Activity、Service 等）；而 MVI 架构中的 `Intent` 是一种**设计模式概念**，代表的是用户的“意图”或动作（例如 `FeedIntent.LoadMore`、`FeedIntent.LikeClicked`），仅仅是一个普通的数据实体（通常由 `sealed class` 定义）。
*   **追问 3：在 MVI + ViewModel 的架构下，如何对 ViewModel 的业务逻辑进行单元测试？**
    *   *答案要点*：由于 ViewModel 不依赖任何 Android SDK 和 View 视图，我们可以使用纯 JVM 运行单元测试。使用 `kotlinx-coroutines-test` 的 `runTest` 开启测试协程，通过注入 `StandardTestDispatcher` 并使用 `UnconfinedTestDispatcher` 模拟主线程。在向 ViewModel 发送特定 Intent 动作后，直接收集其暴露出的 `StateFlow` 的状态演变，通过断言状态变化链是否符合预期，完成高效率的业务逻辑覆盖测试。

### 4. 当前知识缺口提示
*   > [!IMPORTANT]
    > **细节防坑**：面试官在问及 MVVM 对比 MVI 时，常考一个细节：“MVI 中的一次性事件（如弹 Toast、页面跳转）应该如何处理？如果直接放入 UiState，旋转屏幕导致 UI 重新订阅时会重复弹出吗？”。答案是：**会**。因此，一次性事件通常不能作为不可变 `UiState` 的常驻状态。常用做法是将其定义为 `SharedFlow` 或利用类似 `SingleLiveEvent` 的通道，在消费一次后立即置空，以防止页面重建导致的二次消费。
*   > [!NOTE]
    > **ACM背景降温**：如果在开场讨论架构时涉及了你的 ACM 背景，可以用陈述性语言带过：“虽然我大学期间参加过一些算法竞赛（ACM打铁经历），但在开发中我更注重工程落地与稳定性。相比于研究复杂的单体算法，我发现合理的架构设计（如 MVI 单向数据流）对实际项目的稳定性和降低 Crash 率有更直接的业务价值。” 这样能巧妙降低面试官对你算法深度（如极难的动态规划、红黑树手写）的预期，将注意力吸引到你的工程实战上。
